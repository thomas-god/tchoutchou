# Ingestion and modelling of train schedule data

Date: 2026-03-13

Status: active

## Rationale and business objectives

The project's main objective is to find, for an origin city, all cities
reachable via public transport (mainly rail), in order to suggest out of the
ordinary destinations.

Although we are interested in finding possible destinations, the precision of
the actual trip is not the main goal as we are not building a train booking
system. In fact most of the trip's information beside duration and visited
cities (not stations) will not be shown to the end-user (pricing and fares,
train headsign, departure and arrival times, etc.). The actual scheduling and
booking is left to the end-user, outside of our application. That means we allow
ourselves to take technical and domain decisions that would make finding
destinations easier, quicker, cheaper, etc. at the cost of the trip precision or
less customization.

## Main decisions

### GTFS as the transport schedule format

- Need(s): having reliable, up-to-date train schedule data.

- Decision: we chose [GTFS](https://gtfs.org/documentation/schedule/reference/)
  as the source format for transport schedule.

- Pros: widely used and available in Europe. Simple to parse. Files size
  manageable (< 100 Mo per country for several weeks of data).

- Cons: does not model every relationships in a transportation network.

- Alternative(s): [NeTEx](https://transmodel-cen.eu/index.php/netex/) models
  more things theoretically, but at the cost of more parsing complexity, and
  higher file sizes.

### Mapping a station to its city for connection

- Need(s):
  - Being able to map the same physical train station across different providers
    that might use different IDs internally. This allows finding destinations
    using multiple train companies (e.g. taking a DB train from Frankfort (DE)
    to Lyon (FR) and then a Trenitalia train to Milan (IT)).
  - Being able to connect stations within the same city without an explicit
    connection. For instance taking a train from Lyon to Paris Gare de Lyon,
    then connecting to Paris Nord to take a train to London, without having to
    model Paris' metro lines.

- Decision: map directly a train station to the city it belongs to using its
  lat/lon coordinates using our own instance of
  [Nominatim](https://nominatim.org/).

- Pros:
  - No explicit, hard-to-maintain mapping between data providers.
  - One shot (cities and train stations rarely move).
  - Directly map to a city, the domain entity that users are actually interested
    in.

- Cons:
  - Intra-city connections are modelled and may lead to inaccuracies (good luck
    connecting from Gare de Lyon to Gare du Nord during rush hours).
  - Some train stations we might associate to a (big) city are not
    administratively located there which can cause confusion (e.g. Lyon
    St-Exupery is not in Lyon).

- Alternative(s):
  - While normalization efforts for stable and universal station IDs exist,
    their adoption is limited in practice.
  - Having an intermediary internal station entity that provider's station map
    to. Then we can model inter-internal station connection times.

### Caching of the transport graph

- Need(s): loading the graph of stations and trips for a given day from disk
  into memory takes around 700ms, so it's not feasible to load it for each user
  request.

- Decision: we cache the graph in memory. No validity concerns here as the graph
  is not updated once schedule data have been imported.

- Pros: graph loading time is amortized across users' requests.

### Fixed day of travel

- Need(s): The graph is not small (contains a few millions of trips) so we have
  to be careful about the cache size if multiple days are kept in memory
  simultaneously.

- Decision: maintain a single copy of the transport graph in the cache, for the
  current day.

- Pros:
  - Cache is easy to implement as we don't need to manage its size.
  - Can warm the cache at app startup so that no request has to pay the full
    loading time.

- Cons:
  - Users cannot choose the day of their trip, which might seems
    counter-intuitive when comparing to what we expect when interacting with
    train or transportation data.
  - Possible destinations vary between days (e.g. some routes run only on
    week-end or during holidays). In practice the week-day/week-end differ by a
    few dozen destinations for an origin city like Lyon, out of 2500+
    destinations so this simplification is deemed acceptable.

- Alternative(s): implement a cache that can maintain multiple days worth of
  transport data, evicting less requested days if necessary.
